# 基于 OpenAI Codex 的本地编码代理实验平台 README / 需求文档

## 1. 项目名称

Codex Local Agent Lab（基于 OpenAI Codex 的本地编码代理实验平台）

---

## 2. 项目背景

本项目用于实现一个面向教学展示与实验验证的“本地编码代理实验平台”。
平台以 **OpenAI Codex CLI / 本地编码代理工作流** 为参考对象，不直接复刻 OpenAI 官方产品，而是实现一个 **可在本地运行、可视化展示、可管理任务、可执行代码操作、可记录会话与测试结果** 的实验型平台。

该平台的目标不是做成完整商业产品，而是做成一个：

(1) 便于课程实验演示；
(2) 便于说明“本地编码代理”的工作机制；
(3) 便于后续继续扩展；
(4) 便于使用 Codex 按模块逐步开发完成。

---

## 3. 项目目标

### 3.1 总体目标

构建一个本地 Web 平台，用户可以通过浏览器发起编码任务，平台在本地工作区中调度“代理执行器”完成以下能力：

(1) 读取项目代码；
(2) 展示文件树与文件内容；
(3) 基于任务描述生成执行计划；
(4) 执行文件修改；
(5) 执行命令；
(6) 展示 diff、日志、测试结果；
(7) 记录会话历史；
(8) 支持审批机制；
(9) 支持基础配置管理。

### 3.2 项目定位

这是一个 **教学/实验平台**，不是官方 Codex 的替代品。

因此实现时应优先保证：

(1) 功能闭环完整；
(2) 界面清晰；
(3) 模块边界清晰；
(4) 便于演示；
(5) 便于后续写实验报告与展示系统流程。

## 4. 推荐技术栈

> 该部分是建议，不是绝对限制。若 Codex 认为有更优替代，可在不破坏目标的前提下调整，但需保持项目结构清晰。

### 4.1 前端

推荐：

(1) Next.js 14+
(2) TypeScript
(3) Tailwind CSS
(4) shadcn/ui 或其他轻量组件库
(5) React Query / SWR（二选一）

### 4.2 后端

推荐：

(1) Next.js API Routes 或独立 Node.js + Express / Fastify
(2) TypeScript
(3) Prisma（如使用数据库）
(4) Zod 用于请求校验

### 4.3 数据存储

推荐优先级：

(1) SQLite：适合本地实验平台，易部署；
(2) PostgreSQL：如果后续想扩展多人协作再切换。

### 4.4 本地代理执行层

推荐：

(1) Node.js child_process / execa 执行本地命令；
(2) simple-git 操作 Git；
(3) diff 库展示文件修改前后差异；
(4) fs/promises 读写项目文件；
(5) chokidar（可选）做目录监听。

### 4.5 认证

第一版可不做复杂认证。

推荐：

(1) 单机实验版：无需登录；
(2) 若需要：只做本地管理员登录。

## 5. 核心业务场景

平台至少要支持以下场景：

### 场景 1：创建一个代理任务

用户输入自然语言任务，例如：

- “分析当前项目结构并给出重构建议”
- “修复测试失败的问题”
- “为某个模块补充单元测试”

平台需要：

(1) 保存任务；
(2) 创建会话；
(3) 进入任务执行页面；
(4) 展示任务状态。

### 场景 2：浏览本地工作区

用户可以看到：

(1) 项目目录树；
(2) 文件内容；
(3) 被修改文件列表；
(4) 当前 Git 状态。

### 场景 3：查看代理执行过程

平台需要以可视化方式展示：

(1) 当前任务；
(2) 执行计划；
(3) 正在执行的命令；
(4) 命令输出；
(5) 文件变更；
(6) 是否触发审批；
(7) 最终结果。

### 场景 4：执行审批

当代理准备执行高风险操作时，例如：

(1) 删除文件；
(2) 批量修改文件；
(3) 执行 shell 命令；
(4) 执行测试；

平台应暂停执行，并由用户点击：

- 允许
- 拒绝

### 场景 5：查看测试结果

用户可以看到：

(1) 测试命令；
(2) 测试输出；
(3) 是否通过；
(4) 失败摘要；
(5) 最近一次测试时间。

### 场景 6：查看历史会话

用户可以回看：

(1) 历史任务；
(2) 历史执行日志；
(3) 历史 diff；
(4) 历史测试结果；
(5) 历史审批记录。

## 7. 核心功能需求

### 7.1 任务管理

必须支持：

(1) 创建任务；
(2) 查看任务详情；
(3) 更新任务状态；
(4) 删除任务；
(5) 重新运行任务。

任务状态建议：

- pending
- planning
- waiting_approval
- running
- testing
- completed
- failed
- rejected

### 7.2 会话管理

每个任务对应一个会话。

会话需要记录：

(1) 用户输入；
(2) 执行计划；
(3) 每一步操作；
(4) 每一步输出；
(5) 审批记录；
(6) 最终结果。

### 7.3 文件管理

平台必须支持：

(1) 读取指定工作区内文件；
(2) 展示目录树；
(3) 修改文件内容；
(4) 记录文件修改历史；
(5) 展示文件 diff；
(6) 禁止访问工作区外路径。

### 7.4 命令执行

平台必须支持：

(1) 执行白名单命令；
(2) 记录命令开始时间和结束时间；
(3) 记录 stdout / stderr；
(4) 支持超时控制；
(5) 支持中断执行；
(6) 支持审批后再执行。

第一版建议只支持以下命令族：

(1) npm / pnpm / yarn
(2) node
(3) pytest / python
(4) git status / git diff / git add / git restore
(5) ls / dir / pwd

### 7.5 审批机制

平台必须实现最简审批机制。

以下动作默认需要审批：

(1) 删除文件；
(2) 批量修改超过 N 个文件；
(3) 执行 shell 命令；
(4) 执行测试命令；
(5) Git 写操作。

审批记录需要保存：

(1) 审批动作；
(2) 触发时间；
(3) 审批人；
(4) 审批结果；
(5) 备注。

### 7.6 测试执行

平台必须支持测试执行与结果记录。

至少包含：

(1) 运行默认测试命令；
(2) 显示测试日志；
(3) 显示通过/失败状态；
(4) 记录最近一次测试结果；
(5) 关联到任务详情。

### 7.7 Git 集成

平台必须支持基础 Git 信息展示：

(1) 当前分支；
(2) 工作区变更文件；
(3) diff 预览；
(4) 最近一次提交信息（可选）；
(5) 回滚当前文件修改（可选）。

## 8. 非功能需求

### 8.1 安全性

必须满足：

(1) 所有文件访问必须限制在工作区目录内；
(2) 所有命令执行必须经过白名单校验；
(3) 高风险操作必须审批；
(4) 前端不能直接执行系统命令；
(5) 后端必须做参数校验；
(6) 所有关键操作必须记录日志。

### 8.2 可维护性

必须满足：

(1) 前后端目录清晰；
(2) API 层、服务层、数据层分离；
(3) 类型定义集中管理；
(4) 公共组件复用；
(5) 关键模块要有注释。

### 8.3 可扩展性

设计时要预留扩展点：

(1) 后续接入真实 LLM；
(2) 后续接入多工作区；
(3) 后续接入多用户；
(4) 后续接入 MCP 工具；
(5) 后续接入更多审批策略。

### 8.4 可用性

必须满足：

(1) 页面布局清晰；
(2) 状态变化可见；
(3) 错误提示明确；
(4) 长时间任务支持 loading 与日志滚动展示；
(5) 测试结果和 diff 要容易阅读。

## 12. API 需求

推荐 REST API。

### 12.1 任务管理

(1) `GET /api/tasks`
获取任务列表

(2) `POST /api/tasks`
创建任务

请求体示例：

```json
{
  "title": "修复测试失败",
  "description": "分析项目并修复 failing tests",
  "workspacePath": "/workspace/demo-project",
  "autoRunTests": true,
  "approvalEnabled": true
}
```

(3) `GET /api/tasks/:id`
获取任务详情

(4) `DELETE /api/tasks/:id`
删除任务

(5) `POST /api/tasks/:id/rerun`
重新运行任务

### 12.2 工作区

(1) `GET /api/workspace/tree`
获取目录树

(2) `GET /api/workspace/file?path=...`
获取文件内容

(3) `POST /api/workspace/file/update`
更新文件内容

(4) `GET /api/workspace/git-status`
获取 Git 状态

(5) `GET /api/workspace/diff?path=...`
获取某文件 diff

### 12.3 命令执行

(1) `POST /api/commands/run`
执行命令

(2) `POST /api/commands/:id/stop`
中断命令

(3) `GET /api/commands/:id`
查看命令执行详情

### 12.4 审批

(1) `GET /api/approvals/pending`
获取待审批项

(2) `POST /api/approvals/:id/approve`
审批通过

(3) `POST /api/approvals/:id/reject`
审批拒绝

### 12.5 测试

(1) `POST /api/tests/run`
运行测试

(2) `GET /api/tests/:id`
查看测试结果

## 13. 代理执行器设计要求

这是平台最核心的部分。

不要把“真实大模型”作为第一版必需依赖。第一版先实现一个 **可运行的模拟代理执行器**。

### 13.1 模拟代理执行器能力

模拟代理执行器收到任务后，应按以下步骤工作：

(1) 读取任务描述；
(2) 生成一份简单执行计划；
(3) 将计划写入 task_steps；
(4) 按规则执行文件读取、命令执行或测试；
(5) 如遇高风险动作则创建 approval；
(6) 完成后写入 summary。

### 13.2 执行器输出规范

每一步执行都要记录：

(1) 类型；
(2) 标题；
(3) 内容；
(4) 状态；
(5) 时间。

### 13.3 后续扩展

第二阶段可以替换为真实 LLM 驱动执行器：

(1) 接入 OpenAI API；
(2) 根据 prompt 生成计划；
(3) 根据工具调用结果继续推理；
(4) 生成最终摘要。
